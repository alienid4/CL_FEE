# 費用合約控管系統 - Excel 欄位盤點與系統化設計 v2.9

日期：2026-06-13  
來源目錄：`A3.附件`  
定位：本系統不是從零重建資料，而是把既有 Excel 控管方式系統化。

---

## 1. PM 修正方向

目前 HTML 原型容易讓人誤會成「全新系統」。正確方向應改為：

```text
既有 Excel 控管 + PDF 簽呈內容
→ 匯入 / 上傳暫存
→ 欄位對應
→ 系統檢核
→ 人工確認
→ 正式資料
→ Dashboard / 角色作業 / 報表快照
→ 匯出正式 Excel
→ 繳交最後單位
```

第一版不能要求業主放棄 Excel。要先讓系統吃得懂 Excel、保留來源、找出問題、降低人工追蹤負擔。

正式交付原則：最後繳交外部或最後單位的資料仍以 Excel 為準。系統不是取代 Excel，而是讓 Excel 匯入後可被檢核、追蹤、補狀態、保留版本，最後再匯出成可繳交的正式 Excel。

## 4. PDF 簽呈內容盤點

`A3.附件` 內目前有 PDF：

```text
36E100011400471.pdf
36E100011500185.pdf
36E100011500234.pdf
附件七、金控、證券2026年度微軟原廠技術支援服務(Unfied Support)續約採購案簽呈.pdf
```

PM 修正：PDF 不應只是「附件」。它是簽呈內容與決策依據，系統要能把 PDF 關聯到 Case、Signing Case、Contract、Payment、Budget_Check 與 Release_Gate。

### 4.1 PDF 建議欄位

| 系統欄位 | 說明 |
|---|---|
| `document_id` | PDF 文件 ID |
| `case_id` | 關聯 Case |
| `signing_case_id` | 關聯上簽案件 |
| `contract_id` | 如可辨識，關聯合約 |
| `file_name` | 原始檔名 |
| `document_type` | 簽呈 / 請購簽呈 / 核銷文件 / 合約附件 / 其他 |
| `official_doc_no` | 公文 / 簽呈編號，如可辨識 |
| `subject` | 簽呈案名 |
| `vendor_name` | 廠商 |
| `amount` | 簽呈金額 |
| `period_start`, `period_end` | 合約或服務期間 |
| `page_no` | 命中頁碼 |
| `text_excerpt` | 命中段落 |
| `parse_status` | 未解析 / 已解析 / 待人工確認 |
| `confirmed_by`, `confirmed_at` | 人工確認人與時間 |

### 4.2 PDF 系統化流程

```text
PDF 簽呈檔
→ 上傳 / 匯入文件暫存
→ 判斷是否文字型 PDF
→ 可解析：抽文字、建立 document_texts
→ 不可解析：保留檔案與人工摘要，OCR 第二階段
→ 人工確認案名、金額、廠商、期間、公文編號
→ 關聯 Case / Signing Case / Contract / Payment / Budget_Check
→ 主管與處長決策時可下鑽到 PDF 頁碼與命中段落
```

第一版原則：

1. 文字型 PDF 先做全文搜尋與頁碼定位。
2. 掃描 PDF OCR 放第二階段。
3. AI / PDF 解析出的金額不得直接寫正式資料，必須人工確認。
4. 處長決策包要能看到簽呈 PDF 來源，而不只是 Excel 摘要。

## 5. Excel + PDF 系統化流程

```text
A3.附件 Excel / PDF
→ Import Batch
  - 檔名
  - Sheet
  - PDF 頁碼
  - Header Row
  - Row No
  - 匯入時間
  - 匯入人

→ Staging Table
  - 保留 Excel 原欄位
  - 不直接覆蓋正式資料
  - 每列給 source_row_id

→ 欄位對應
  - 費用項目主檔 → cost_items / contracts
  - 預算編列 → budget_lines
  - 次月上簽 → signing_cases
  - 核銷 / 付款明細 → payment_schedules / reimbursements / invoices
  - 續約合約進度 → contract_renewal_tasks
  - 專案追蹤 → projects / project_progress
  - PDF 簽呈 → signing_documents / document_texts / source_references

→ 系統檢核
  - 必填欄位
  - 金額格式
  - 日期格式
  - 合約起迄日
  - 預算內外
  - 是否逾期
  - 合約 / 費用項目 / 專案候選比對
  - PDF 金額與 Excel 金額是否一致
  - PDF 案名與 Excel 案名是否可勾稽

→ 人工確認
  - 助理確認欄位 mapping
  - 承辦補附件與原因
  - 主管確認風險與 Release_Gate
  - 處長只看 Release_Gate 案件
  - 需要時下鑽 PDF 原文與頁碼

→ 正式資料
  - Case
  - Cost Item
  - Contract
  - Payment
  - Budget
  - Reimbursement
  - Invoice
  - Audit Logs
  - Signing Documents

→ Dashboard / 報表快照

→ 匯出正式 Excel
  - 符合最後單位要求欄位
  - 符合最後單位要求欄位順序
  - 維持原範本格式、Sheet、欄名、欄序、公式、合併儲存格與樣式
  - 原範本已有對應欄位時，才帶入系統檢核後的最新處理進度、核銷編號、附件狀態、提醒狀態與主管確認狀態
  - 匯出前檢核必要欄位與格式
  - 原範本沒有的系統追蹤欄位不放入正式繳交 Excel
```

### 5.1 年度預算範本版本管理

`115年預算範本.xlsx` 屬於年度預算正式範本。此類 Excel 可能一年多版，不可用最後上傳檔案直接覆蓋正式資料。

建議流程：

```text
年度預算 Excel 匯入
→ 建立預算版本
→ 系統解析欄位與金額
→ 與目前正式版本比對
→ 顯示新增 / 刪除 / 金額變更 / 負責窗口變更 / 類別變更
→ 助理確認
→ 設為目前正式版本
→ 主管查詢與追溯
→ 依原範本格式匯出正式 Excel
```

版本狀態：

| 狀態 | 說明 |
|---|---|
| 待確認 | 已匯入但尚未成為正式版本。 |
| 目前正式版本 | 主管與報表預設使用的版本。 |
| 歷史版本 | 曾經正式或曾匯入保留查核的版本。 |
| 作廢 | 匯入錯誤或確認不使用的版本，仍保留紀錄。 |

版本追蹤資料留在系統，不放入正式繳交 Excel。正式匯出時只回填原範本既有欄位，並保留公式、合併儲存格與樣式。

## 6. UI 改善方向

四角色頁應改成 Excel-first：

| 角色 | 第一層 | 第二層 | 第三層 | 第四層 |
|---|---|---|---|---|
| 助理 / 管理者 | Excel 匯入批次與錯誤數 | Sheet / Row 清單 | 單列欄位 mapping | 原 Excel 來源與修正紀錄 |
| 承辦 | 我的待補資料 | 待補 Case / Row | 補來源、PDF 簽呈、附件、付款、核銷 | 退回與上傳紀錄 |
| 主管 | 風險摘要 | 預算外 / 合約不明 / 逾期清單 | 單案判斷資料 | Excel 原列、PDF 簽呈、附件、audit logs |
| 處長 | Release_Gate 摘要 | 待決策清單 | 單案決策包 | PDF 簽呈、放行紀錄與報表快照 |

## 7. 不要做歪的地方

1. 不要把 Excel 丟掉，第一版要能承接 Excel。
2. 不要讓使用者重填 Excel 已經有的欄位。
3. 不要把每張 Excel 都硬做成一個系統頁面，要先統一到 Case / Contract / Payment / Budget / Invoice / Audit。
4. 不要直接用匯入資料覆蓋正式資料，必須先暫存、檢核、人工確認。
5. 不要把專案追蹤表強行當費用主檔，它只應在正式 Project_ID 成立時關聯。
6. 不要讓處長看到所有 Excel row，只看 Release_Gate 後的決策包。
7. 不要把 PDF 當一般附件，它是簽呈依據，必須能回到頁碼與內容。
8. 不要只做系統查詢而沒有正式 Excel 匯出，因為最後繳交單位仍需要 Excel。
9. 不要為了系統追蹤而改正式繳交 Excel 格式，正式匯出必須維持原範本。
