## 專案目標

開發「費用合約控管系統」第一版，做出可供主管、業助、承辦人操作的系統本體。

使用者必須可以查案件、看預算、專案、簽呈、合約、請購、付款、發票、附件與舉證狀態。
系統不是單純查表工具，而是費用、專案、簽呈、合約、付款、發票與來源舉證的控管追蹤系統。

第一版以本機 / 開發環境可展示、可驗證、可中斷續跑為目標。
正式 AD、正式 MSSQL、真實 PDF 解析、正式外部系統介接可先預留，不得假裝已完成。

## 第一版必做功能

1. 八項控管看板：預算、專案、簽呈、案件管理、核准、合約、請購、付款、資料檢核。
2. 全文檢索：可搜尋案件、合約、簽呈、付款、發票、廠商、負責人、附件。
3. Dashboard 數字下鑽：每個數字都要能點出明細，例如未完成幾件、多少金額由哪些 Case 組成。
4. Case 360：每個案件要能看到基本資料、時間線、簽呈、合約、付款、發票、附件、舉證。
5. 時間線：顯示預算、專案、簽呈、核准、合約、請購、付款、發票節點，並用顏色表示完成、未完成、不適用、異常、進行中。
6. 處理優先矩陣：讓主管知道哪些案件要先處理，並可點進案件明細。
7. 附件管理：PDF 可作為附件登錄與關聯，但第一版不做 PDF 自動解析。
8. Excel 匯入匯出預留：先做欄位 mapping、匯入批次、暫存與檢核概念，真實複雜解析可第二版深化。
9. 發票、簽呈、合約、付款資料都要有欄位與畫面位置。
10. 廠商視角：可看廠商相關合約、付款、發票、案件。
11. 負責人視角：可看負責人手上案件、未完成、金額與待辦。
12. CMDB 預留：第一版只保留 CMDB 關聯欄位與畫面，不串接 CMDB。
13. 稽核紀錄：重要新增、修改、停用、刪除、匯入確認與狀態變更需保留 audit log 或預留欄位。
14. 來源舉證鏈：資料至少要預留來源檔、Sheet、列號、欄位、原始值、文件或附件關聯。

## 第一版不做

1. 不解析真實 PDF。
2. 不匯入真實電子發票。
3. 不串接 CMDB。
4. 不要求使用者提供敏感資料。
5. 不做正式 AD / SSO 權限。
6. 不做正式營運資安控管，但要保留未來擴充位置。
7. 不把 AI 判斷直接寫入正式金額或正式狀態。
8. 不直接碰正式資料、正式 DB、正式憑證或不可逆部署。

## 權限與角色

第一版需先保留角色與權限模型，但不實作正式 AD / SSO 登入。

第一版至少需支援四種業務角色概念：

1. CIO / 高階主管
   - 可看全公司或全域統計。
   - 可看所有案件、金額、風險、異常與 Dashboard。
   - 以檢視、追蹤、決策為主。

2. 部門主管 / 業助
   - 可看所屬部門或被授權範圍內的案件。
   - 可維護案件、追蹤待辦、確認附件與資料狀態。
   - 可看負責人、廠商、付款與發票狀態。

3. 承辦人 / 負責人
   - 可看自己負責或被指派的案件。
   - 可維護進度、備註、附件、待辦與處理狀態。
   - 可查看與自己案件相關的簽呈、合約、付款、發票與舉證。

4. 系統管理者
   - 可維護基礎資料、匯入設定、欄位 mapping、角色設定與系統規則。
   - 可查看稽核紀錄與資料檢核結果。
   - 不代表一定能任意修改業務金額；重要異動仍需留紀錄。

第一版權限實作方式：

- 可先用前端角色切換、測試帳號或簡化 role 欄位模擬。
- API 與資料模型需保留 `role`、`permission_scope`、`owner_department`、`owner_user` 等概念。
- 重要操作需保留 audit log。
- UI 需能呈現不同角色視角，但不要求正式 AD 驗證。

第二版 AD 整合方向：

- 使用 AD / LDAP / Entra ID 或公司既有 SSO 作為登入來源。
- 以 AD 群組控制系統角色與資料範圍。
- AD 群組可對應系統角色，例如：
  - `AI_FEE_CIO` -> CIO / 高階主管。
  - `AI_FEE_MANAGER` -> 部門主管 / 業助。
  - `AI_FEE_HANDLER` -> 承辦人 / 負責人。
  - `AI_FEE_ADMIN` -> 系統管理者。
- 未來 AD 控權要以後端 API 權限為準，不能只靠前端隱藏按鈕。

## 資料策略

1. 目前只能使用使用者提供的截圖、Excel 樣本、文件描述與假資料。
2. 敏感資料暫時不能提供；不得要求使用者貼真實密碼、token、憑證、完整個資或正式資料。
3. PDF 第一版只當附件與舉證，不抽欄位。
4. 未來第二版再接真實資料、PDF 解析、電子發票解析、CMDB。
5. 公司正式系統未指定 DB 時，預設方向是 Microsoft SQL Server；第一版本機可先用 SQLite，但需保留 MSSQL adapter / 設定擴充點。
6. 回匯與匯入不得直接覆蓋正式資料；必須先暫存、檢核、人工確認，再寫入正式表。

## 開發分級策略

### 小修邊界

可視為小修：

- 測試設定修正
- 明確 UI bug
- 明確 API 小錯誤
- 文案或文件小修
- 不改資料模型、不改權限、不改核心流程的修正

不得視為小修：

- DB schema 變更
- Excel 匯入 / 回匯
- 權限 / AD / 登入
- MSSQL adapter
- 部署流程
- 報表金額邏輯
- 會影響正式資料或資料正確性的行為

### 小修 / 明確 bug

適用：

- 測試設定修正
- 小型 UI bug
- 文案調整
- 明確前端事件 bug
- 明確 API 小錯誤

流程：

- 直接修。
- 直接跑相關測試。
- 不啟動完整多 Agent 流程。
- 回報變更檔案、測試結果、是否需重啟。

### 中型功能

適用：

- 單一模組功能
- 單一 API + UI 切片
- 可局部驗證的功能

流程：

- 使用 Coder + Tester。
- Coder 實作。
- Tester 驗證。
- 必須有測試或可重現驗證。

### 大功能 / 跨模組功能

適用：

- DB schema
- Excel 匯入 / 回匯
- 權限 / AD
- MSSQL
- 部署
- 資料安全
- 報表邏輯
- 跨模組資料一致性

流程：

- 使用 Architect + Coder + Tester + Reviewer。
- Architect 定義切片與風險。
- Coder 實作。
- Tester 驗證。
- Reviewer 檢查風險、範圍、測試、archive 誤用與資料安全。

正式資料、權限、DB、匯入回匯、部署一律保守。
涉及不可逆 migration、正式部署、憑證、正式資料時，必須停下並回報風險與需要確認事項。

## Agent 規則

不要為了形式拖慢小修。
小修可由單一 Codex 直接執行；中型以上才啟動 Coder / Tester 或四 Agent 流程。

若啟動 Agent，請至少包含：

- Architect：定義需求邊界、資料模型、API 架構、設計圖與驗收標準；不直接改檔案。
- Coder：實作程式碼、資料模型、API、前端、seed data、腳本與文件更新；可改檔案。
- Tester：設計與執行測試，檢查首頁、API、搜尋、下鑽、Case 360、附件、CMDB 預留、Dashboard 數字一致性。
- Reviewer：檢查資料正確性、API 相容性、第一版 / 第二版邊界、敏感資料、archive 誤用、版面風格與剩餘風險。

Agent 證據規則：

- 若實際啟動獨立 Agent，必須記錄 sub-agent id、Calling Agent 區塊、工具呼叫紀錄或回傳摘要。
- `.github/agents/*.agent.md` 只算設定，不算實際呼叫證據。
- 若沒有獨立 Agent 呼叫證據，不得在 `docs/agent_run_report.md` 寫「獨立 Agent 已完成」；只能寫「角色模擬」或「獨立 Agent 未啟動」。
- 若環境無法呼叫獨立 Agent，必須明確回報降級方式。

Agent KPI 只在中型以上功能或主要階段結束時更新；小修不必硬寫完整 KPI。

## 提速規則

1. 不做空泛 heartbeat。
2. 長任務才定期回報；回報必須有差異、風險、測試或下一步。
3. 每次回報必須寫「跟上次差異」：
   - 新增了什麼
   - 哪些不變
   - 是否退步
   - 是否新增阻塞
4. 如果沒有新增實作，必須明確寫：「無新增實作，僅狀態維持」，然後立即選擇下一個安全 backlog 推進。
5. 每次只做一個小切片。
6. 不做大範圍重構。
7. 不修改與當前切片無關的檔案。
8. 不用「測試通過」當作自動停止理由；若 backlog 還有安全、可本機驗證的工作，應接續下一個切片。
9. 若使用者明確要求只回報狀態、暫停或停止，才停止開發。

## v2.0 產出優先 Watchdog 補充規則（2026-07-05）

本補充用於提高 Codex 實際產出比例，避免過度消耗在空泛 heartbeat、長篇狀態管理或假多 Agent 回報。

### 目前有效主版本

- 主提示詞以 `docs/一次性開發提示詞_v1.9.md` 為準。
- 本節視為 v2.0 行為補丁；若與舊版 5-10 分鐘長回報規則衝突，以本節為準。
- `docs/一次性開發提示詞_v1.7.md` 保留為歷史版本，不作為最新主提示詞。

### 執行節奏

- 採用「產出優先 + 輕量監控」模式。
- 每個功能切片目標時間為 60-90 分鐘。
- 小修 / 明確 bug：主模型直接修 + 跑測試，不啟動完整多 Agent。
- 中型功能：啟動 Coder + Tester。
- 涉及 DB schema、權限、匯入確認寫入、正式部署、資料安全：啟動 Architect + Coder + Tester + Reviewer。
- 每輪只做一個明確切片，不做無關重構。

### 30 分鐘 Watchdog

- 每 30 分鐘只做輕量 watchdog，不寫空泛長文。
- 若切片仍在進行，只需簡短回報：
  - 目前動作。
  - 跟上次差異。
  - 阻塞原因。
  - 下一步。
- 若切片完成、有實質變更、測試結果更新、或遇到阻塞，才同步更新：
  - `docs/AI開發進度.md`
  - `docs/agent_run_report.md`
- 若連續一次以上無新增實作且無阻塞，不能只回報，必須立即選擇下一個安全 backlog 推進，或明確說明需要使用者決策。

### 確保仍在執行的證據

每次 watchdog 或切片完成回報，至少提供一項可驗證證據：

- 新增 / 修改的檔案。
- 剛執行的測試命令與結果。
- 目前 active slice 名稱。
- 已啟動或已關閉的 Agent ID。
- 已重啟或確認的 runtime endpoint。

若沒有任何證據，必須寫：

```text
無新增實作，僅狀態維持；下一步立即推進：<安全 backlog 名稱>
```

### 文件更新策略

- 不再每個 heartbeat 都重寫長篇文件。
- 文件只在以下情況更新：
  1. 切片完成。
  2. 測試結果有變化。
  3. 完成度有變化。
  4. 阻塞狀態有變化。
  5. Agent 狀態有變化。
- 文件內容必須短而可追蹤，不寫空泛敘述。

### 完成度規則

- `整體專案完成度：約 XX%` 必須是整個 AI_FEE 產品的真實估計，不是單一切片完成度。
- 若要補充局部進度，另列 `本階段完成度`。
- 完成度只有在功能、測試、文件或可驗收能力實質增加時才上調。
