# 權限與 AD 整合規劃

## 決策

第一版先保留角色與權限模型，不實作正式 AD / SSO。

第二版再接 AD / LDAP / Entra ID 或公司既有 SSO，並用 AD 群組控制系統角色與資料範圍。

## 第一版角色

### 1. CIO / 高階主管

可看：

- 全域 Dashboard。
- 所有案件。
- 所有金額彙總。
- 異常、逾期、待確認。
- 負責人與廠商統計。

主要用途：

- 決策。
- 追蹤風險。
- 查看整體資源與費用狀況。

### 2. 部門主管 / 業助

可看：

- 所屬部門或被授權範圍內案件。
- 負責人工作量。
- 待辦、附件、簽呈、合約、付款、發票狀態。

可做：

- 維護案件資料。
- 補附件。
- 更新處理狀態。
- 追蹤待辦。
- 匯入或確認資料。

### 3. 承辦人 / 負責人

可看：

- 自己負責或被指派案件。
- 自己案件的 Case 360。
- 相關簽呈、合約、付款、發票、附件與舉證。

可做：

- 更新進度。
- 補備註。
- 上傳或登錄附件。
- 處理待辦。

### 4. 系統管理者

可做：

- 維護角色設定。
- 維護欄位 mapping。
- 維護匯入批次。
- 維護狀態規則。
- 查看稽核紀錄。
- 管理系統參數。

限制：

- 系統管理者不代表可以無痕修改業務金額。
- 重要異動仍需 audit log。

## 第一版實作原則

第一版可以先用簡化方式：

- 前端角色切換。
- 測試帳號。
- role 欄位。
- permission_scope 欄位。
- owner_department 欄位。
- owner_user 欄位。

第一版必須保留：

- 角色模型。
- API 權限擴充點。
- audit log。
- 不同角色視角。

第一版不做：

- AD 登入。
- SSO。
- LDAP 查詢。
- 正式權限 middleware。
- 正式部門權限同步。

## 第二版 AD 整合方向

AD 群組對應建議：

| AD 群組 | 系統角色 | 說明 |
|---|---|---|
| AI_FEE_CIO | CIO / 高階主管 | 全域檢視 |
| AI_FEE_MANAGER | 部門主管 / 業助 | 部門或授權範圍 |
| AI_FEE_HANDLER | 承辦人 / 負責人 | 個人案件 |
| AI_FEE_ADMIN | 系統管理者 | 系統設定 |

第二版需支援：

- AD / SSO 登入。
- 群組轉角色。
- 部門屬性轉資料範圍。
- API 權限檢查。
- 前端按鈕權限。
- 操作稽核。
- 權限測試案例。

## 權限控制原則

- 權限必須由後端 API 控制，不能只靠前端隱藏按鈕。
- 所有重要查詢都要套資料範圍。
- 所有重要異動都要寫 audit log。
- 若權限不足，API 需回傳明確錯誤。
- 匯出 Excel 也必須受權限控制。

## 驗收標準

第一版：

- 可以切換或模擬四種角色視角。
- 畫面能呈現不同角色關注的資料。
- 資料模型保留未來 AD 欄位與權限擴充點。
- 操作紀錄有 audit log。

第二版：

- 可用 AD / SSO 登入。
- AD 群組可正確轉換為系統角色。
- API 權限與資料範圍正確。
- 不同角色只能查到授權資料。
- 匯出與下鑽也受權限控管。
