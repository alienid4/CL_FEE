# 核心功能與驗收清單

本文定義八項控管看板延伸出的十二項核心功能。後續開發、測試、Review 都必須以本清單作為驗收依據。

## 系統定位

本系統不是單純查表工具，而是給主管、業助、承辦與使用者操作的費用、專案、簽呈、合約、請購、付款、發票控管系統。

核心目標是讓使用者可以從任何角度追查一件事：

- 這是哪個案件或專案。
- 預算、簽呈、合約、請購、付款、發票走到哪裡。
- 哪些完成、未完成、不適用、異常。
- 金額從哪裡來，是否可被舉證。
- 誰負責，誰卡住，下一步是誰。
- 可從總數、金額、負責人、廠商、月份、狀態一路下鑽到明細。

## 一、Case 360 詳情頁

每一個案件必須有完整詳情頁，作為所有查詢與點擊的最終落點。

必要內容：

- 基本資料：Case ID、案件名稱、年度、部門、負責人、廠商、來源。
- 流程狀態：預算、專案、簽呈、核准、合約、請購、付款、發票。
- 關聯資料：相關簽呈、合約、付款排程、發票、附件、匯入來源。
- 金額彙總：預算金額、合約金額、已付款、待付款、發票金額、差異。
- 狀態判斷：完成、未完成、不適用、異常、待確認、逾期。
- 舉證來源：Excel 檔名、Sheet、列號、欄位、原始值、匯入時間。

驗收標準：

- 從搜尋結果、Dashboard 數字、時間線節點、處理優先泡泡都能進入同一個 Case 360。
- 使用者不用切換多個頁面就能知道案件全貌。

## 二、待辦、交辦、追蹤

系統必須能把資料問題轉成可處理任務。

必要內容：

- 待確認、待補件、待付款、待開票、待簽核、逾期等待辦。
- 可指派負責人、到期日、優先順序、處理說明。
- 可記錄處理結果與完成時間。
- 可從 Case、負責人、部門、主管視角查看。

驗收標準：

- Dashboard 上的「未完成 N 件」可點出是哪 N 件。
- 主管可以看到誰手上有多少待辦與逾期項目。

## 三、提醒通知

系統必須支援到期與異常提醒。

必要內容：

- 付款到期提醒。
- 發票應開未開提醒。
- 合約到期提醒。
- 簽呈或附件缺漏提醒。
- 金額異常或資料不一致提醒。
- 提醒對象、提前天數、提醒頻率可設定。

驗收標準：

- 每一筆提醒都能回到對應 Case 或證據資料。
- 主管可以看到提醒是否已處理。

## 四、權限與角色

系統需依角色呈現可看的資料與可做的操作。

必要角色：

- CIO：看全域、金額、風險、異常與統計。
- 部門主管 / 業助：看所屬案件、待辦、流程進度、資料維護。
- 承辦 / 使用者：看自己負責或被指派的案件與待辦。
- 系統管理者：維護資料來源、規則、角色、匯入設定。

驗收標準：

- 不同角色看到的 Dashboard、清單、操作按鈕不同。
- 重要修改需要留下操作紀錄。

## 五、版本與異動紀錄

所有重要資料需保留修改歷程。

必要內容：

- 匯入版本。
- 人工修改前後值。
- 修改人、修改時間、修改原因。
- 狀態變更紀錄。
- 金額變更紀錄。

驗收標準：

- 任一 Case 可追查資料從哪個版本來。
- 可比較新舊 Excel 匯入造成的差異。

## 六、資料品質檢核

系統必須主動標出資料問題。

必要檢核：

- 必填欄位缺漏。
- 金額格式錯誤。
- 日期格式錯誤。
- 重複 Case、重複合約、重複付款。
- 合約金額與付款排程不一致。
- 已付款超過合約金額。
- 發票金額與付款金額不一致。
- 找不到關聯案件。

驗收標準：

- 匯入後可看到錯誤、警告、可人工確認三種等級。
- 每個錯誤都能定位到來源檔案、Sheet、列與欄。

## 七、附件、PDF、簽呈管理

系統必須能管理與案件相關的文件證據。

必要內容：

- 附件上傳與下載。
- PDF、簽呈、合約、報價單、驗收文件分類。
- 文件與 Case、合約、付款、發票關聯。
- 文件缺漏提醒。
- 重要文件可標記頁碼、段落、摘要或證據文字。

驗收標準：

- 使用者查一個 Case 時，可以看到所有相關文件。
- 付款、發票、簽呈欄位都能反查支撐文件。

## 八、報表中心

系統需支援管理層與作業層報表。

必要報表：

- 年度預算彙總。
- 合約與付款彙總。
- 月份付款預測。
- 月份應開發票清單。
- 未完成 / 異常 / 逾期清單。
- 負責人工作量。
- 廠商金額與案件彙總。

驗收標準：

- 報表數字可下鑽到明細。
- 可匯出 Excel。
- 報表數字與 Case 360 的明細合計一致。

## 九、廠商視角

系統必須可以用廠商作為查詢主軸。

必要內容：

- 廠商基本資料。
- 廠商相關案件。
- 合約清單。
- 付款排程。
- 已付款、待付款。
- 發票開立狀況。
- 異常與逾期。

驗收標準：

- 搜尋廠商名稱可看到完整往來紀錄。
- 可回答「這個廠商今年有多少案子、多少金額、哪些未付款」。

## 十、狀態規則管理

系統狀態不可只靠人工填寫，需有可解釋規則。

必要內容：

- 完成、未完成、不適用、異常、待補、待確認、逾期的判斷規則。
- 不同功能節點可有不同規則。
- 規則結果需可被人工覆核。
- 人工覆核需留下理由。

驗收標準：

- 時間線顏色與 Dashboard 數字來自同一套規則。
- 使用者可以知道為什麼某案件被判定為異常或未完成。

## 十一、關聯建立、合併、拆分

匯入資料常不乾淨，系統需支援人工建立與修正關聯。

必要內容：

- Case 與預算、簽呈、合約、付款、發票關聯。
- 同一案件多筆來源資料可合併。
- 一筆來源資料可拆成多個案件或付款項目。
- 無法自動關聯時進入待確認清單。

驗收標準：

- 系統能處理 Excel 欄位名稱不一致、案名不完全相同、同一案多來源的情境。
- 合併或拆分後仍保留原始來源證據。

## 十二、Excel 式操作模式

因使用者熟悉 Excel，系統需提供接近 Excel 的批次檢視與操作。

必要內容：

- Excel 匯入。
- Excel 匯出。
- 欄位排序、篩選、凍結表頭。
- 批次編輯。
- 批次確認。
- 匯出目前篩選結果。
- 匯出完整資料。
- 匯出含證據與系統判斷的資料。

驗收標準：

- 使用者可以用 Excel 範本匯入資料。
- 匯出資料可回到業務作業使用。
- 匯入與匯出欄位需有版本控管。

## 橫向核心能力

以下能力不是單一頁面，而是所有功能都要具備。

### 全文檢索

搜尋必須跨模組：

- 案件名稱。
- Case ID。
- 簽呈 ID。
- 合約 ID。
- 付款 ID。
- 發票資訊。
- 廠商。
- 負責人。
- 附件摘要。
- Excel 原始欄位。

驗收標準：

- 搜尋「XX專案」可以看到所有相關資料。
- 搜尋結果可依案件、合約、付款、發票、文件分組。
- 搜尋結果可進入 Case 360。

### 數字下鑽

所有 Dashboard 數字都必須可點擊。

例子：

- 未完成 8 件：點擊後列出 8 件。
- 1,500 萬：點擊後列出組成金額的案件或付款。
- 6 月應開發票：點擊後列出廠商、發票、付款狀態。

驗收標準：

- 每個數字都有明細來源。
- 明細合計必須等於 Dashboard 數字。

### 時間線

時間線是主管理解案件進度的視覺入口。

節點：

- 預算。
- 專案。
- 簽呈。
- 核准。
- 合約。
- 請購。
- 付款。
- 發票。

顏色：

- 綠色：完成。
- 灰色：不適用或未開始。
- 藍色：進行中或目前卡點。
- 黃色：待補、待確認、需注意。
- 紅色：異常、逾期、需主管處理。

驗收標準：

- 每個節點可點進細項。
- 時間線狀態與 Case 360、Dashboard 數字一致。

### 處理優先矩陣

處理優先矩陣用來讓主管快速知道先處理哪件事。

必要內容：

- 依緊急程度、金額影響、等待天數、主管介入需求分區。
- 泡泡大小可代表金額或件數。
- 泡泡顏色可代表狀態。
- 泡泡可點入 Case 360 或篩選後清單。

驗收標準：

- 主管能回答「今天最該處理哪幾件」。
- 點擊泡泡能看到組成案件與原因。

### 舉證鏈

每個系統判斷都要能說明根據。

必要內容：

- 來源檔名。
- Sheet。
- 列號。
- 欄位。
- 原始值。
- 匯入時間。
- 修改人。
- 修改後值。
- 判斷規則。
- 關聯文件或附件。

驗收標準：

- 使用者看到金額或狀態時，可以追到來源。
- Review 可以檢查資料正確性，不只看 UI 顯示。

## Excel 範本需求

後續開發需要使用者提供或確認各模組 Excel 範本：

- 年度預算。
- 專案清單。
- 簽呈 / 上簽。
- 合約。
- 請購。
- 付款排程。
- 發票。
- 資料檢核或附件清單。

若範本尚未固定，系統需先支援欄位對應設定，避免每次 Excel 格式微調就需要改程式。

## 開發驗收原則

每新增一個功能，必須同時檢查：

- 是否可搜尋。
- Dashboard 數字是否可下鑽。
- 是否可回到 Case 360。
- 是否有來源舉證。
- 是否支援角色權限。
- 是否可匯出 Excel。
- 是否有測試案例。
- 是否有 Reviewer 檢查資料正確性與 API 相容性。
