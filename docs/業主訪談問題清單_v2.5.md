## 2. 用案例 1 確認紅線流程

案例：中華電信板橋機櫃電力費調整  
假設：中華電信通知因台電漲價，每月增加 3,000 元電力費。

請業主確認：

| 判斷點 | 目前設計 | 業主確認 |
|---|---|---|
| 是否合約相關？ | 是，建立候選 Contract_ID |  |
| 是否特殊合約關聯？ | 是，Contract_Relation = 費用調整 |  |
| 是否正式專案？ | 否，Project_ID 空白 |  |
| 是否產生付款？ | 是，Payment_Schedule 每月 3,000，12 期 |  |
| 是否預算外？ | 是，年度影響 36,000 |  |
| 是否需處長放行？ | 暫定需要 Release_Gate |  |
| 收到帳單後如何處理？ | 建立 Expense_Claim，再建立 Invoice |  |
| 是否需要發票分攤？ | 預設不需要，除非一張發票含多筆費用 |  |

要問業主的關鍵句：

```text
這個案例如果發生在你們單位，紅線是不是這樣走？
哪一步不對？
哪一步應該由誰確認？
這種金額和情境需不需要處長看？
```

## 6. 資料來源確認

請業主確認第一版要接哪些來源，哪些先手動。

| 來源 | 第一版處理方式 | 業主確認 |
|---|---|---|
| Excel 費用合約整合檔 | 匯入暫存，人工確認後進正式資料 |  |
| 次月預計上簽清單 | 匯入成 Case / Signing Plan |  |
| PDF 簽呈內容 | 以簽呈文件管理，記錄案號、主旨、金額、期間、頁碼、文字摘錄與人工確認 |  |
| 其他合約附件 PDF | 先存附件與來源，不做掃描 OCR |  |
| Email 通知 | 第一版可手動建立來源紀錄 |  |
| AI 金額解析 | 只做建議，不直接寫正式金額 |  |
| 既有合約系統編號 | 可先候選勾稽，人工確認 |  |

要問業主：

```text
第一版哪些資料一定要匯入？
哪些可以先人工填？
哪些來源一定要能追溯？
PDF 簽呈裡哪些欄位一定要抓：簽呈案號、主旨、廠商、金額、期間、付款條件、決策結論？
PDF 是否大多是文字型，還是有掃描影像需要 OCR？
處長決策包是否必須顯示 PDF 頁碼與文字摘錄，讓主管或處長能回查原文？
```

## 8. PM 提醒

不要讓討論變成「畫面要放哪個按鈕」。現在要先確認規則。

也不要把 PDF 當成普通附件。這些 PDF 是簽呈內容，會影響金額、期間、付款條件與放行依據，必須可追溯到頁碼與原文。

如果業主開始討論 UI，可以拉回來問：

```text
這個畫面之後可以調整。
我現在先確認：這種案件到底需不需要建立 Contract_ID / Project_ID / Payment / Release_Gate？
```

判斷規則穩了，畫面、API、資料表才有依據。
