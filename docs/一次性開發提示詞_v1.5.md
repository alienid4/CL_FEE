# 一次性開發指令 v1.5

你是系統分析師、架構師、開發者、測試者與 Reviewer。

請依照本專案既有文件與程式碼，開發「費用合約控管系統」第一版。

我已確認以下範圍，請開始第一版開發，直到完成設計圖、開發、測試、Review 與部署。
若非阻斷問題，不要停下來問我；一般細節請自行採保守合理假設。

## 第一版目標

做出可供主管、業助、承辦人操作的系統本體，讓使用者可以查案件、看預算、專案、簽呈、合約、付款、發票、附件與舉證狀態。

## 第一版必做功能

1. 八項控管看板：預算、專案、簽呈、案件管理、核准、合約、請購、付款、資料檢核。
2. 全文檢索：可搜尋案件、合約、簽呈、付款、發票、廠商、負責人、附件。
3. Dashboard 數字下鑽：每個數字都要能點出明細，例如未完成幾件、多少金額由哪些 Case 組成。
4. Case 360：每個案件要能看到基本資料、時間線、簽呈、合約、付款、發票、附件、舉證。
5. 時間線：顯示預算、專案、簽呈、核准、合約、請購、付款、發票節點，並用顏色表示完成、未完成、不適用、異常、進行中。
6. 處理優先矩陣：讓主管知道哪些案件要先處理，並可點進案件明細。
7. 附件管理：PDF 可作為附件登錄與關聯，但第一版不做 PDF 自動解析。
8. Excel 匯入匯出預留：先做欄位 mapping 與匯入批次概念，第二版再做真實批次解析。
9. 發票、簽呈、合約、付款資料都要有欄位與畫面位置。
10. 廠商視角：可看廠商相關合約、付款、發票、案件。
11. 負責人視角：可看負責人手上案件、未完成、金額與待辦。
12. CMDB 預留：第一版只保留 CMDB 關聯欄位與畫面，不串接 CMDB。

## 第一版不做

1. 不解析真實 PDF。
2. 不匯入真實電子發票。
3. 不串接 CMDB。
4. 不要求我提供敏感資料。
5. 不做正式 AD/SSO 權限。
6. 不做正式營運資安控管，但要保留未來擴充位置。

## 權限與角色

第一版需先保留角色與權限模型，但不實作正式 AD / SSO 登入。

第一版至少需支援四種業務角色概念：

1. CIO / 高階主管
   - 可看全公司或全域統計。
   - 可看所有案件、金額、風險、異常與 Dashboard。
   - 以檢視、追蹤、決策為主。

2. 部門主管 / 業助
   - 可看所屬部門或被授權範圍內的案件。
   - 可維護案件、追蹤待辦、確認附件與資料狀態。
   - 可看負責人、廠商、付款與發票狀態。

3. 承辦人 / 負責人
   - 可看自己負責或被指派的案件。
   - 可維護進度、備註、附件、待辦與處理狀態。
   - 可查看與自己案件相關的簽呈、合約、付款、發票與舉證。

4. 系統管理者
   - 可維護基礎資料、匯入設定、欄位 mapping、角色設定與系統規則。
   - 可查看稽核紀錄與資料檢核結果。
   - 不代表一定能任意修改業務金額；重要異動仍需留紀錄。

第一版權限實作方式：

- 可先用前端角色切換、測試帳號或簡化 role 欄位模擬。
- API 與資料模型需保留 `role`、`permission_scope`、`owner_department`、`owner_user` 等概念。
- 重要操作需保留 audit log。
- UI 需能呈現不同角色視角，但不要求正式 AD 驗證。

第二版 AD 整合方向：

- 使用 AD / LDAP / Entra ID 或公司既有 SSO 作為登入來源。
- 以 AD 群組控制系統角色與資料範圍。
- AD 群組可對應系統角色，例如：
  - `AI_FEE_CIO` -> CIO / 高階主管。
  - `AI_FEE_MANAGER` -> 部門主管 / 業助。
  - `AI_FEE_HANDLER` -> 承辦人 / 負責人。
  - `AI_FEE_ADMIN` -> 系統管理者。
- AD 屬性或群組可用來判斷部門、可視資料範圍與管理權限。
- 第二版需補登入、登出、權限 middleware、API 權限檢查與操作稽核。

限制：

- 第一版不得假裝已完成正式 AD 權限控管。
- 第一版只做角色模型、畫面視角、權限欄位與未來擴充點。
- 未來 AD 控權要以後端 API 權限為準，不能只靠前端隱藏按鈕。

## 資料策略

1. 目前只能使用我提供的截圖、Excel 樣本、文件描述與假資料。
2. 敏感資料暫時不能提供。
3. PDF 第一版只當附件與舉證，不抽欄位。
4. 未來第二版再接真實資料、PDF 解析、電子發票解析、CMDB。

## 需求控管

請用系統分析師角度檢查我是否漏掉第一版必要功能。

若你主動補充需求，必須分成：

1. 第一版必要：若不做會影響第一版可用性、資料正確性、驗收或後續擴充。
2. 第一版可選：做了更好，但不影響第一版驗收。
3. 第二版建議：需真實資料、敏感資料、外部系統、正式權限或較完整流程才適合做。

若你判斷有第一版必要功能，請直接納入本次開發。
第一版可選與第二版建議只記錄在 docs，不要等待我確認，也不要納入本次開發。
未經我明確同意，不得把「第一版可選」或「第二版建議」納入本次開發範圍。
若你認為某功能會影響資料正確性、驗收或後續擴充，請先列為風險或第一版必要，並說明原因。

## 多 Agent 開發要求

請務必使用多 agent 模式；若目前環境無法啟動真正獨立 agent，請至少以明確分段方式模擬 Architect、Coder、Tester、Reviewer 的產出與審查流程，並在回報中標示。

請至少包含以下角色：

### Architect

- 負責閱讀文件、定義第一版需求邊界、資料模型、API 架構、設計圖與驗收標準。
- 負責判斷哪些是第一版必要、第一版可選、第二版建議。
- 只能提出設計、測試範圍與風險，不直接改檔案。

### Coder

- 負責實作程式碼、資料模型、API、前端、seed data、腳本與文件更新。
- 只有 Coder 可以改檔案。
- 若修改資料模型、API 或流程，必須同步更新 docs 與設計圖。

### Tester

- 負責設計與執行測試，補足測試案例。
- 必須檢查首頁、API、搜尋、下鑽、Case 360、附件、CMDB 預留、Dashboard 數字一致性。
- 若發現問題，回報給 Coder 修正。

### Reviewer

- 負責檢查資料正確性、API 相容性、第一版/第二版邊界、敏感資料是否誤入 Git、版面風格是否一致、剩餘風險。
- Reviewer 不直接改檔案，只提出修正建議。

協作規則：

1. Architect 先輸出需求邊界、設計圖、資料模型與測試範圍。
2. Coder 依 Architect 結論實作。
3. Tester 跑測試並補測案例。
4. Reviewer 檢查風險、資料正確性、API 相容性與範圍是否失控。
5. 若 Tester 或 Reviewer 發現第一版必要問題，交由 Coder 修正。
6. 只有 Coder 可以改檔案。
7. 每個階段都要更新 `docs/AI開發進度.md`。
8. 最後回報四個角色的結論。

## Git 檢查與版本控管

1. 開發前請先檢查本機是否已安裝 Git。
2. 若未安裝 Git，請提醒我安裝，或在可行且安全的情況下協助安裝。
3. 若專案尚未初始化 Git，請執行 `git init`。
4. 請建立或更新 `.gitignore`，避免將敏感資料、PDF、Excel、DB、venv、log、快取檔、壓縮檔納入版本控管。
5. 請設定 `git config core.quotepath false`，讓中文檔名可讀。
6. 請先用 `git status` 檢查有哪些檔案會被納入。
7. 不要把敏感原始資料 commit。
8. 每完成一個重大階段，請建立 Git commit，commit message 需清楚說明本階段內容。
9. 若無法 commit，請在 `docs/AI開發進度.md` 記錄原因、目前 `git status`、下一步。
