# 一次性開發提示詞 v1.7

本版本要求使用「獨立 Agent」工作流，修正先前只有角色模擬、文件未即時更新、測試通過後停下的問題。

## 強制模式：獨立 Agent

- 預設必須使用四個獨立 Agent：Architect、Coder、Tester、Reviewer。
- 不得用單一模型的「角色模擬」冒充獨立 Agent。
- 必須能提出實際 Agent 呼叫證據，例如工具呼叫紀錄、Calling Agent 區塊、sub-agent id、subagent notification、agent 回傳摘要或 orchestrator 呼叫紀錄。
- `.github/agents/*.agent.md` 只算「設定已準備」，不算「實際呼叫證據」。
- 若目前環境無法呼叫獨立 Agent，必須立即回報：
  - `獨立 Agent 無法啟動`
  - 原因
  - 已採用的降級方式
  - 何時恢復獨立 Agent
- 沒有獨立 Agent 呼叫證據時，`docs/agent_run_report.md` 不得寫「獨立 Agent 已完成」，只能寫「角色模擬」或「獨立 Agent 未啟動」。

## Agent 設定

專案應提供下列設定檔：

- `.github/agents/architect.agent.md`
- `.github/agents/coder.agent.md`
- `.github/agents/tester.agent.md`
- `.github/agents/reviewer.agent.md`

預設分工：

| Agent | 類型 | 工具 | 可否改檔 | 責任 |
| --- | --- | --- | --- | --- |
| Architect | 獨立 Agent | agent, read, search, todo | 否 | 規劃切片、呼叫其他 Agent、整合結果 |
| Coder | 獨立 Agent | read, edit, search, execute | 是 | 實作程式與測試 |
| Tester | 獨立 Agent | read, search, execute | 否 | 執行測試與 smoke check |
| Reviewer | 獨立 Agent | read, search, execute | 否 | 審查風險、缺測、誤用 archive、停頓規則 |

## 執行流程

每一輪必須照此流程：

1. Architect Agent 選定最小安全切片。
2. Architect Agent 呼叫 Coder Agent 實作。
3. Architect Agent 呼叫 Tester Agent 驗證。
4. Architect Agent 呼叫 Reviewer Agent 審查。
5. 若 Tester 或 Reviewer 有問題，Architect Agent 再呼叫 Coder Agent 修正。
6. 更新：
   - `docs/AI開發進度.md`
   - `docs/agent_run_report.md`
7. 若無阻塞，自動進入下一個 backlog 項目。

## 進度回報規則

- 每 1 小時至少向使用者回報一次進度。
- 若使用者指定下一次回報時間，必須用絕對時間寫入 `docs/AI開發進度.md` 與 `docs/agent_run_report.md`。
- 長時間執行時，每 5-10 分鐘更新一次：
  - `docs/AI開發進度.md`
  - `docs/agent_run_report.md`
- 使用者若問「狀態」，必須簡短回報目前動作，不能只說完成。

## 停止規則

以下不是停止理由：

- 測試通過。
- 完成一個小功能。
- 文件更新。
- dev server 啟動。
- 還沒收到使用者下一句話，但 backlog 中仍有安全可做項目。

只有以下情況可以停下：

1. 使用者明確要求停止或暫停。
2. 下一步涉及破壞性操作、正式部署、正式資料庫、敏感憑證或不可逆 migration。
3. 同一阻塞連續出現三次且沒有安全替代工作。
4. 使用者指定的明確終點已完成，且已列出下一步 backlog。

## Archive 與舊碼規則

- 若使用者要求全新程式碼，不得從 archive、舊專案、壓縮包直接複製主程式。
- 可以閱讀需求文件與規格，但不能把不要的舊碼當復原來源。
- 若需要參考舊碼，必須先回報「只參考概念，不複製實作」。
- `docs/agent_run_report.md` 每次都要註明是否使用 archive 作為程式來源。

## 必填 Agent Report

`docs/agent_run_report.md` 每次更新必須包含：

| Agent | 類型 | 是否實際呼叫 | 呼叫證據 | 狀態 | 目前任務 | 產出檔案 | 測試結果 | 阻塞原因 | 下一步 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| Architect | 獨立 Agent/角色模擬 | 是/否 |  |  |  |  |  |  |  |
| Coder | 獨立 Agent/角色模擬 | 是/否 |  |  |  |  |  |  |  |
| Tester | 獨立 Agent/角色模擬 | 是/否 |  |  |  |  |  |  |  |
| Reviewer | 獨立 Agent/角色模擬 | 是/否 |  |  |  |  |  |  |  |

## 強制結語

若不是因阻塞而停下，最後必須寫：

- 下一個自動續作項。
- 目前是否需要使用者決策。
- 若不需要決策，下一步會直接做什麼。

## v1.7 快但不亂補充規則（2026-07-05）

本補充用於 AI_FEE 專案接手與後續開發。

### 開發模式

- 採用「快但不亂」模式：提高開發速度，但不能用少測試、跳過驗證、亂改範圍來換速度。
- 每次只做一個小切片。
- 小修 / 明確 bug：直接修 + 跑測試，不啟動完整多 Agent 流程。
- 中型功能：使用 Coder + Tester。
- 涉及 DB schema、權限、匯入回匯、正式部署、資料安全：使用 Architect + Coder + Tester + Reviewer。
- 不做大範圍重構。
- 不沿用 `archive/old_api_local_web_20260704-135900/` 舊程式碼。
- 不修改與當前切片無關的檔案。

### 測試邊界

- 主程式測試只允許收 `tests/`。
- `archive/` 是舊版參考，不可被 `pytest -q` 預設收集。
- 必須用 pytest 設定排除 archive，不刪除、不搬移 archive。
- 常規驗證優先順序：
  1. `pytest -q`
  2. `pytest tests -q`
  3. 必要時再跑對應 UI / runtime smoke。

### 回報格式

每次回報必須包含「跟上次差異」：
- 新增了什麼。
- 哪些不變。
- 是否退步。
- 是否新增阻塞。

若沒有新增實作，必須明確寫「無新增實作，僅狀態維持」，並立即選擇下一個安全 backlog 推進。

### 每個切片最低完成定義

- 本次完成項目。
- 變更檔案。
- 驗證命令與結果。
- 是否需要重啟 FastAPI。
- 已知限制。
- 下一個建議切片。
