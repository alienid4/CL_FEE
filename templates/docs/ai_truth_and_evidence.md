# AI Truth And Evidence Rules

這份文件是 AI 協作底線。AI 不是不能推理，而是不能把推理偽裝成事實。

## 核心規則

| 類型 | 可以怎麼說 | 不可以怎麼說 |
| --- | --- | --- |
| 事實 | 根據 log 顯示服務未啟動 | 我猜服務壞了 |
| 假設 | 我推測可能是權限問題，需要檢查檔案 owner | 一定是權限問題 |
| 建議 | 建議先跑 read-only 診斷 | 直接改 production |
| 缺資料 | 目前沒有足夠證據判斷 | 隨便補一個原因 |

## 必須查證的項目

- 公司環境與主機狀態。
- 版本、release、patch 是否存在。
- 部署是否成功。
- 資料筆數、資料品質、資料來源。
- 錯誤原因與 root cause。
- 權限、帳號、token、API key、連線狀態。
- 會影響正式資料或正式服務的操作。

## 證據優先順序

1. 程式碼與設定。
2. 自動化測試結果。
3. API / health / ready endpoint。
4. 應用 log、audit log、job result。
5. 使用者提供的去敏化診斷輸出。
6. 截圖與人工描述。

## 證據不足時

請提供一次性診斷指令，且必須：

- read-only。
- 不印出密碼、token、private key。
- 不印出完整個資或敏感清單。
- 優先輸出到 `/tmp` 或本機 temp。
- 用 hash、count、status、summary 取代原始敏感值。
