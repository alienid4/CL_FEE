# 費用合約控管系統 - 欄位確認問題與系統建議 v3.0

版本：v3.0  
日期：2026-06-13  
用途：帶去跟業主確認欄位規則。這份文件採「我方先給系統建議，業主只確認業務判斷」的方式。

---

## 1. 討論原則

不要問業主：「DB 要不要拆表？」  
這是系統設計問題，我方應直接給建議。

要問業主：「這個欄位的業務意義是什麼？哪個來源比較準？誰可以改？什麼情況算完成？」  
這些才是業主必須判斷的事。

我方建議如下：

```text
Excel 全欄位先被系統接住
→ 原始資料進匯入暫存區
→ 每個欄位都有去處
→ 正式資料拆成 Case / Contract / Budget / Payment / Document / Audit 等表
→ 人工確認後才寫入正式資料
→ 保留 Excel row、Sheet、檔名、PDF 頁碼與簽呈內容來源
```

也就是：Excel 如果總共有 150 欄，系統設計要能說明 150 欄各自去哪裡，但正式 DB 不應做成一張 150 欄大表。

---

## 2. 欄位去處的我方建議

| 欄位類型 | 我方建議 | 要不要問業主 |
|---|---|---|
| 穩定主檔欄位 | 進正式主檔，例如費用項目、廠商、統編、合約期間、負責人 | 問欄位定義與主來源 |
| 交易 / 明細欄位 | 拆成明細表，例如付款年度、月份、金額、期別、計費期間 | 不問拆表，只問業務規則 |
| 月份欄位 | `1月` 到 `12月` 不建議做 12 個正式欄位，建議轉成 `year + month + amount` | 問報表是否仍要看 1-12 月格式 |
| 文件 checklist 欄位 | 轉成文件需求與文件狀態，例如費用核銷申請書、付款憑證、合約書 | 問哪些文件必須上傳 |
| PDF 簽呈欄位 | 轉成簽呈文件、文字摘錄、頁碼、來源追溯 | 問 PDF 內哪些欄位必須抓 |
| 備註 / 差異說明 | 保留，但不要當流程判斷主欄位 | 問哪些備註其實有固定狀態意義 |
| 計算欄位 | 系統計算，例如未付款金額、完成度、合約天數、逾期與否 | 問計算規則，不問是否存欄位 |
| 舊欄位 / 不再使用欄位 | 原始匯入保留，正式資料可標記不使用 | 問是否仍有人使用 |

## 3. 各 Excel 來源的我方定位

| 來源檔 | 我方建議定位 | 要問業主確認 |
|---|---|---|

| `費用合約整合(舊).xlsx` | 歷史資料與欄位對照，不建議覆蓋新版主檔 | 哪些 Sheet 還在用，哪些只留歷史 |
| `2026次月預計上簽項目.xlsx` | 上簽計畫來源，匯入成 Case / Signing Plan | 每月 Sheet 是否就是當月正式上簽計畫 |
| `架構部續約合約進度.xlsx` | 續約任務看板來源，關聯合約到期與續約狀態 | 續約狀態文字是否有固定定義 |
| `資訊架構部處級專案進度追蹤總表 (2).xlsx` | 專案進度來源，只在正式 Project_ID 成立時關聯，不當費用主檔 | 哪些費用案件需要連到專案 |
| `金控專案_20260323.xlsx` | 金控 / 跨子公司合作標籤與處長關注條件 | 哪些合作案需要進處長決策包 |
| PDF 簽呈檔 | 簽呈內容與決策依據，不是一般附件 | 要抓哪些欄位與是否需頁碼追溯 |

## 4. 欄位確認時的建議答案

| 問題 | 我方建議 | 只需要業主決定的部分 |
|---|---|---|
| Excel 欄位是否全部進 DB？ | 全部先進匯入暫存與來源紀錄；正式 DB 只存有業務意義的欄位 | 哪些欄位仍有業務用途 |
| 是否做一張 150 欄大表？ | 不建議。會變成網頁版 Excel，後續流程、權限、報表都會卡住 | 不需要問業主 |
| 欄位同名但來源不同怎麼辦？ | 建立來源優先順序與欄位對照表，不自動覆蓋 | 哪一份檔案是主來源 |
| 金額欄位很多怎麼辦？ | 拆成預算金額、合約總金額、付款金額、核銷金額、發票金額、議價費用 | 每個金額是否含稅、幣別、取數來源 |
| 日期欄位很多怎麼辦？ | 拆成合約期間、計費期間、申請日、付款月份、簽呈月份、專案起迄 | 日期的業務定義 |
| `1月` 到 `12月` 要怎麼處理？ | 正式資料轉成月明細；報表輸出仍可呈現 1-12 月欄位 | 報表是否要維持舊格式 |
| 備註欄怎麼處理？ | 保留原文，但若備註其實代表狀態，應拉成正式欄位 | 哪些備註文字其實有固定流程意義 |
| 狀態欄怎麼處理？ | 建議統一狀態字典，例如未啟動、進行中、待補件、待核准、已完成、作廢 | 各 Excel 現有狀態文字如何對應 |
| 負責人 / 窗口怎麼處理？ | 建議統一成 owner_user、owner_group、handler_user | 誰是正式責任人，誰只是聯絡窗口 |
| 文件欄位怎麼處理？ | 轉成文件 checklist，不只是一堆欄位 | 哪些文件必備、哪些可選 |
| PDF 怎麼處理？ | 作為簽呈內容，需存頁碼、文字摘錄、關聯案件與人工確認 | PDF 內必抓欄位 |
| AI 解析欄位可否自動寫入？ | 不建議。AI / PDF 解析只進建議區，人工確認後才寫正式資料 | 誰有權確認 |
| Excel 回匯要不要保留？ | 建議第一版保留，因為使用者仍會用 Excel 溝通 | 哪些報表或清單要回匯 |
| 舊檔案資料要不要匯入？ | 建議匯入為歷史或參考，不直接蓋新版主檔 | 歷史要保留幾年 |
| Project 表要不要併進費用主檔？ | 不建議。專案進度是另一條資料線，只用 Project_ID 關聯 | 哪些案件算正式專案 |

## 5. 必須問業主的欄位問題

以下才是會議要問的問題。

### 5.1 來源優先順序

```text
如果同一個案件、合約或費用項目在多份 Excel 都出現，以哪份為準？
我方建議：
1. 目前作業中的新版整合檔為主。
2. 舊版整合檔只做歷史與比對。
3. 次月上簽檔只做上簽計畫，不覆蓋費用主檔。
請業主確認是否同意。
```

### 5.2 欄位是否正式使用

```text
每個欄位請確認：
1. 正式使用
2. 只做參考
3. 只保留原始來源
4. 不再使用

我方建議：
不要在會議逐欄討論 150 欄，先用欄位群組確認。
有爭議的欄位再逐欄確認。
```

### 5.3 金額定義

```text
請確認各金額欄位是否含稅、是否同幣別、是否可能有分攤。

我方建議拆成：
- budget_amount：預算金額
- contract_amount：合約總金額
- negotiated_amount：議價費用
- payment_amount：付款金額
- reimbursement_amount：核銷金額
- invoice_amount：發票金額
- allocated_amount：分攤金額
```

### 5.4 日期定義

```text
請確認各日期欄位的業務意義。

我方建議拆成：
- contract_start_date / contract_end_date：合約期間
- billing_period_start / billing_period_end：計費期間
- reimbursement_request_date：費用核銷申請日
- payment_year / payment_month：付款年度與月份
- signing_month：預計上簽月份
- project_start_date / project_end_date：專案期間
```

### 5.5 狀態與燈號

```text
請確認 Excel 裡的「狀態、處理進度、合約進度、燈號」各自代表什麼。

我方建議：
不要讓每張表各自一套狀態，正式系統統一成狀態字典。
原 Excel 狀態文字保留在來源欄位。
```

### 5.6 文件與 PDF 簽呈內容

```text
請確認 PDF 簽呈中哪些欄位必須抓：
- 簽呈案號
- 主旨 / 案名
- 廠商
- 金額
- 期間
- 付款條件
- 決策結論
- 頁碼
- 文字摘錄

我方建議：
第一版先做文字型 PDF 搜尋與頁碼定位。
掃描 OCR 放第二階段。
PDF / AI 抓出的金額只能進建議區，不能直接覆蓋正式資料。
```

### 5.7 權限與責任人

```text
請確認誰可以建立、修改、確認、退回、作廢。

我方建議：
- 助理 / 管理者：匯入、欄位 mapping、資料健檢、退回補件
- 承辦：補資料、上傳文件、更新核銷 / 發票
- 主管：確認風險、預算外、合約不明、逾期
- 處長：只看 Release_Gate 後的決策包
```

### 5.8 Excel 回匯

```text
最後繳交單位仍需要 Excel，本系統第一版必須支援正式 Excel 匯出。

需向業主確認：
1. 最後繳交單位要求的 Excel 欄位。
2. 欄位順序、欄名、月份格式、金額格式。
3. 原範本中的公式、合併儲存格與樣式是否都必須保留。
4. 原範本既有欄位中，哪些欄位可由系統回填，例如處理進度、核銷編號、附件狀態、提醒狀態。

我方建議：
Excel 是正式交付格式，系統是控管、檢核、追蹤與匯出工具。
標準流程為：Excel 匯入 → 系統暫存 / 檢核 / 人工確認 → 系統追蹤與補狀態 → 匯出正式 Excel → 繳交最後單位。
正式匯出必須維持原格式，不新增系統欄位、不改欄序、不改 Sheet。系統追蹤欄位留在 DB、稽核紀錄或內部查核匯出。
若未來允許 Excel 回匯修正，不能直接覆蓋正式資料，必須先進回匯暫存與差異比對。
```

## 6. 不需要問業主、我方直接建議的事

| 議題 | 我方直接建議 |
|---|---|
| DB 是否正規化 | 要。不要做 150 欄大表。 |
| Excel 原始欄位是否保留 | 要。完整保留在匯入暫存與來源 JSON。 |
| 最後正式交付格式 | Excel。系統需維持原範本格式正式匯出。 |
| 系統追蹤欄位是否加進正式 Excel | 不加。留在 DB、稽核紀錄或內部查核匯出。 |
| AI 是否可直接寫正式資料 | 不可。必須人工確認。 |
| PDF 是否只是附件 | 不是。PDF 是簽呈內容與決策依據。 |
| 1-12 月是否做正式 12 欄 | 不建議。正式資料做月明細，報表再轉 12 欄。 |
| 專案進度是否併入費用主檔 | 不建議。用 Project_ID 關聯。 |
| 處長是否看全部 Excel row | 不建議。只看符合 Release_Gate 的決策包。 |
| 舊檔是否覆蓋新檔 | 不建議。舊檔作歷史與比對。 |

## 7. 會議說法建議

可以這樣跟業主說：

```text
我們不是要請您設計資料庫。
資料庫怎麼拆表、欄位怎麼正規化，這部分我們會負責。

今天要請您確認的是三件事：
第一，哪份 Excel 是正式來源。
第二，這些欄位的業務意義與責任人是什麼。
第三，PDF 簽呈裡哪些內容必須成為決策依據。

我們的建議是：
Excel 全欄位先保留，正式資料再拆成案件、合約、預算、付款、核銷、文件與稽核。
所以不會漏欄位，也不會把系統做成一張很難維護的大 Excel。
```
